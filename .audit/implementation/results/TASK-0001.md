## Resumen de la implementación

El 404 no se debía a archivos faltantes: `eventos/ical/encuentro-nacional-2026.ics` existe y responde
200. Las referencias eran **relativas** y, bajo la política canónica sin barra final, resolvían contra
la raíz del sitio (`/ical/…`) en lugar de `/eventos/ical/…`. Se sustituyeron por rutas **absolutas**
en ambas páginas y se declaró el tipo MIME de calendario.

Se verificó antes de tocar nada que `assets/js/calendar.js` (ruta protegida) resuelve el atributo con
`new URL(value, window.location.href)` y deriva el nombre de archivo con `split('/').pop()`: ambas
operaciones funcionan igual con ruta absoluta, por lo que **no requirió cambios**. Un solo cambio de
atributo repara las **dos** opciones rotas del diálogo, porque `appleLink.href` y `downloadLink.href`
comparten la misma URL resuelta.

## Nota sobre `.htaccess` — actualizada 2026-07-20

Durante la implementación, `.htaccess` estaba excluido de Git (entrada heredada de la plantilla
WordPress del `.gitignore`) y se desplegaba solo dentro del ZIP manual. **El propietario decidió
versionarlo el mismo día**, tras verificar que el archivo no contiene credenciales, autenticación ni
listas de IP: es configuración de servidor observable desde fuera.

En consecuencia, `git revert` **sí** es una vía de reversión válida para este archivo, y los
procedimientos de rollback de TASK-0001, TASK-0004, TASK-0008 y TASK-0013 se actualizaron para
reflejarlo. Se conserva el matiz importante: revertir en Git no cambia producción por sí solo, porque
el despliegue es por ZIP manual (ADR 0015).

Pendiente para el corte a WordPress: WP regenera su propio bloque `# BEGIN WordPress` /
`# END WordPress`; las reglas propias deben quedar fuera de él. Anotado como comentario en el
`.gitignore`.
